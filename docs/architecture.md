# Архитектура

## Контур сервисов

- `apps/api` принимает Telegram webhook, сохраняет историю, достает контекст, вызывает AI и применяет policy guard.
- `apps/worker` обрабатывает фоновые задачи: уведомления менеджерам и embeddings базы знаний.
- `apps/admin` дает ручной контроль над каталогом, knowledge base и handoff.
- `PostgreSQL` хранит каталог, лиды, тикеты, диалоги и промпты.
- `Redis` держит очереди BullMQ.
- `OpenAI` используется для диалогов и embeddings, но локальная разработка не ломается без ключа благодаря fallback-режиму.

## Главный runtime flow

1. Telegram присылает update на `POST /webhooks/telegram`.
2. API создает или обновляет `Conversation`.
3. API сохраняет пользовательское сообщение.
4. API достает активный проект, релевантные квартиры и документы знаний.
5. AI возвращает `AIDecision`.
6. `ResponsePolicyService` проверяет, что нет недопустимых обещаний и неподтвержденных квартир.
7. API сохраняет ответ, синхронизирует `Lead` или `SupportTicket`.
8. Если нужен человек, создается job в Redis.
9. Worker шлет уведомление менеджеру.

## Принципы безопасности продукта

- Модель не имеет прямого права обещать то, чего нет в таблицах и контексте.
- В коде есть post-processing guard, который режет неподтвержденные рекомендации.
- Все реальные диалоги для обучения должны анонимизироваться перед попаданием в eval/train корпус.
